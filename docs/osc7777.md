# OSC 7777 -- Wezmux Agent Status Protocol

OSC 7777 is a custom escape sequence that lets programs (typically AI coding agents) send structured status updates to Wezmux. These updates drive the real-time agent status display in the [sidebar](sidebar.md).

## Format

```
ESC ] 7777 ; EVENT ; DATA BEL
```

Or using the `ST` (String Terminator) form:

```
ESC ] 7777 ; EVENT ; DATA ESC \
```

Where `EVENT` is one of the defined event types and `DATA` is the event payload.

## Events

### `status` -- update agent state

Changes the agent's status indicator on the sidebar card.

```
\x1b]7777;status;working\x07
\x1b]7777;status;idle\x07
\x1b]7777;status;needs_input\x07
```

| Value | Sidebar indicator | Meaning |
|-------|-------------------|---------|
| `working` | Spinning animation (orange) | Agent is actively processing |
| `idle` | Solid circle (green) | Agent finished or is waiting for a new prompt |
| `needs_input` | Triangle (yellow/accent) | Agent needs user attention (approval, question, etc.) |

### `message` -- update preview text

Sets the message shown on the workspace card below the agent status line.

```
\x1b]7777;message;Refactoring auth module\x07
```

When the agent transitions from `working` to `idle`, the last message is preserved as a preview so users can see what the agent was last doing.

### `tool` -- report current tool

Reports which tool the agent is currently using. Cached for future display.

```
\x1b]7777;tool;Bash\x07
```

### `subagents` -- update running subagent count

Reports how many subagents are currently running within this agent session (e.g. Claude Code's parallel Agent tool invocations). The sidebar shows a line like "⤷ 3 agents running" below the status indicator when the count is greater than zero.

```
\x1b]7777;subagents;3\x07
\x1b]7777;subagents;0\x07
```

The count is tracked externally via a temp file (`/tmp/wezmux-subagents-<session_id>`) because each hook invocation is a separate process. The `on-subagent-start.sh` and `on-subagent-stop.sh` hooks are the sole authority: they increment/decrement the count under a mkdir-based lock and emit the OSC. The `on-stop.sh` hook intentionally does NOT reset the count — background subagents may still be running when the main agent's turn ends.

### `clear` -- reset agent status

Resets the agent's status to idle and clears transient state. The last working message is preserved.

```
\x1b]7777;clear\x07
```

## Status transition rules

Wezmux applies guards to prevent flickering from race conditions between hooks:

- **`needs_input` is sticky for 3 seconds** -- an `idle` event within 3 seconds of `needs_input` is ignored (guards against the Stop hook firing right after the PreToolUse/AskUserQuestion hook)
- **`working` always overrides `needs_input`** -- when the user answers and the agent resumes, the transition is immediate
- **Entering `working` clears the message** -- avoids showing stale text from a previous task

## How it differs from OSC 9

| | OSC 9 | OSC 7777 |
|---|-------|----------|
| **Purpose** | Generic toast notification | Structured agent status |
| **Format** | `ESC ] 9 ; body BEL` | `ESC ] 7777 ; event ; data BEL` |
| **State** | Stateless (fire-and-forget) | Stateful (drives a status machine) |
| **Sidebar effect** | Increments unread badge, shown as fallback preview | Updates agent status indicator, message, and tool |

Both can be emitted together. The Wezmux hook scripts emit OSC 7777 for structured status and OSC 9 for the notification store, so the sidebar gets both real-time status and notification counts.

## Quick test

To test from any terminal pane:

```bash
# Show "working" spinner on the sidebar card
printf '\033]7777;status;working\007'

# Set a message
printf '\033]7777;message;Running tests...\007'

# Back to idle
printf '\033]7777;status;idle\007'
```
