  <action>
    This task sets up the WezTerm fork in the current (mostly empty) wezmux repo.

    Step 1 - Fork via GitHub CLI (per D-01):
    Run `gh repo fork wez/wezterm --clone=false` to create the fork on GitHub under the user's account.
    If the fork already exists, `gh repo fork` will report that and succeed.

    Step 2 - Pull fork content into the current repo (per D-05, fork from main HEAD):
    The current wezmux repo already has a .git and .planning directory. Add the user's GitHub fork as `origin`:
    ```
    git remote remove origin 2>/dev/null
    git remote add origin https://github.com/$(gh api user --jq .login)/wezterm.git
    git fetch origin
    git checkout -B main origin/main
    ```
    This replaces the current repo content with the full WezTerm source from main HEAD.

    IMPORTANT: The .planning directory and CLAUDE.md must be preserved. Before the checkout:
    1. Copy `.planning/` and `CLAUDE.md` and `DESIGN.md` to a temp location
    2. Perform the checkout
    3. Copy them back and commit: `git add .planning CLAUDE.md DESIGN.md && git commit -m "chore: add wezmux planning artifacts"`

    Step 3 - Configure upstream remote (per D-02, FORK-02):
    ```
    git remote add upstream https://github.com/wez/wezterm.git
    git fetch upstream
    ```

    Step 4 - Verify remotes:
    ```
    git remote -v
    ```
    Expect output showing:
    - `origin` pointing to `github.com/{user}/wezterm.git`
    - `upstream` pointing to `github.com/wez/wezterm.git`

    Per D-02: Stay on `main` branch. The `wezmux/dev` branch will be created in Phase 2 when code changes begin.
    Per D-03: No renaming in Phase 1. Keep all WezTerm naming as-is.
  </action>
  <verify>
    <automated>git remote -v | grep -q upstream && git remote -v | grep -q origin && echo "PASS: both remotes configured" || echo "FAIL"</automated>
  </verify>

    - `git branch --show-current` outputs `main`

    - `ls Cargo.toml` succeeds (WezTerm source is present)

<task type="auto">
  <name>Task 2: Build and verify WezTerm runs on macOS</name>
  <files>Cargo.toml</files>
  <read_first>
    - Cargo.toml (workspace root manifest, verify it exists and has [workspace])
    - CLAUDE.md (build commands: cargo build --package wezterm, cargo clippy)
  </read_first>
  <action>
    This task verifies the fork builds and the resulting binary launches correctly on macOS.

    Step 1 - Debug build (per D-04, use debug for daily dev):
    ```
    cargo build --package wezterm
    ```
    This targets just the GUI binary, skipping ancillary tools like wezterm-mux-server.
    Expected: build succeeds, binary at `target/debug/wezterm` (or `target/debug/wezterm-gui`).

    Step 2 - Clippy lint check:
    ```
    cargo clippy --package wezterm-gui -- -D warnings
    ```
    Expected: exits 0 with no warnings.

    Step 3 - Verify binary exists and is executable:
    ```
    ls -la target/debug/wezterm-gui
    file target/debug/wezterm-gui
    ```
    Expected: Mach-O 64-bit executable arm64 (or x86_64 on Intel).

    Step 4 - Smoke test launch:
    Launch the binary briefly to confirm it starts without crashing:
    ```
    timeout 5 ./target/debug/wezterm-gui start 2>&1 || true
    ```
    The binary will either open a window (success) or crash with an error.
    If running in a headless environment (no display), the command may fail with a display error — that is acceptable. The key check is that the binary is a valid executable and does not have missing dylib errors.

    Step 5 - Run existing tests:
    ```
    cargo test -p wezterm-gui --lib 2>&1 | tail -20
    ```
    Confirm existing tests pass (we have not modified anything).

    Note: The build will take significant time (10-30 minutes for first build of the full workspace).
    Use `cargo build --package wezterm 2>&1 | tail -5` to capture just the final status.
  </action>
  <verify>
    <automated>test -f target/debug/wezterm-gui && echo "PASS: binary exists" || echo "FAIL: binary not found"</automated>
  </verify>
  <acceptance_criteria>
    - `cargo build --package wezterm` exits with code 0
    - `target/debug/wezterm-gui` exists and is a Mach-O executable (`file target/debug/wezterm-gui` contains "Mach-O")
    - `cargo clippy --package wezterm-gui -- -D warnings` exits with code 0
    - `cargo test -p wezterm-gui --lib` exits with code 0 (or reports 0 test failures)
  </acceptance_criteria>
  <done>
    WezTerm builds from source on macOS without errors (FORK-01). The binary is a valid Mach-O executable. Clippy passes with no warnings. Existing tests pass. The fork is a clean, unmodified WezTerm instance (FORK-03).
  </done>
